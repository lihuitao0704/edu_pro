# 单端口部署与统一认证前端设计

## 目标

将 Vue 前端作为 FastAPI 的生产静态资源由 `main.py` 在 `8000` 端口统一提供；用户通过一次 `python main.py` 启动后端 API 与 Web 界面。提供统一的登录/注册入口，登录后依据后端签发 JWT 中的角色自动进入对应工作台。仅允许新注册账号成为客户。

## 范围与边界

- 生产入口固定为 `http://127.0.0.1:8000`（可由 Uvicorn 参数覆盖），不再要求用户单独启动 Vite。
- Vite 只作为本地开发热更新工具，继续将 `/api` 代理到 `http://127.0.0.1:8000`；生产构建使用相对 `/api` 地址。
- 客户可自助注册；投顾、客户经理、风控专员、管理员账号仅能由现有管理/初始化流程维护。
- 权限的最终依据是服务端 JWT 与 API 鉴权；前端角色信息仅用于导航与默认落地页，不用于授予权限。
- 不引入 Nginx、外部身份提供商或多租户能力。

## 架构与启动流程

`main.py` 保持 FastAPI API 服务，并在存在 `frontend/dist` 时挂载构建资源、提供 SPA history fallback。根路径和前端路由均返回 Vue 应用，`/api/*` 请求继续由 FastAPI 路由处理。`main.py` 的直接启动入口绑定 `8000`。

前端构建仍由 `npm run build` 生成 `frontend/dist`。为避免运行时端口耦合，浏览器 API 客户端的默认地址保持 `/api`；开发时由 Vite 代理，生产时由同源 FastAPI 处理。项目文档将提供“构建一次、`python main.py` 启动”的标准命令，以及可选的前端开发命令。

## 认证与身份路由

认证入口包含登录和注册两种状态，可相互切换且共享品牌、表单控件、错误提示、加载状态与响应式布局。

登录向 `POST /api/auth/login` 提交用户名和密码。认证成功后，前端安全地保存 access token 与已登录用户的最小展示信息，随后按角色选择首页：客户进入智能对话、投顾进入顾问工作台、客户经理进入业务操作、风控专员进入风险管理、管理员进入知识库。

路由守卫要求受保护页面具有会话，并仅允许访问该角色声明可见的菜单路由；非法或过期会话清空本地会话信息并回到登录页。API 返回 401 时也应清空会话。受限 API 的服务端授权保持现有 JWT 中间件与角色校验。

## 注册流程

注册页收集真实姓名、用户名、手机号（可选）、密码与确认密码。前端执行必填、长度、密码确认与用户名格式验证；服务端继续作为唯一的数据完整性与重复用户名判断来源。

注册请求使用现有 `POST /api/auth/register`。服务端无论客户端提交何种字段，均创建 `CUSTOMER` 用户且不写入员工角色。成功后客户端使用相同凭据立即调用登录接口，取得 JWT 后跳转至客户首页。注册失败会在表单内展示可操作的错误消息，不泄露额外账户信息。

## 前端质量基线

- 使用现有 Vue 3、TypeScript、Pinia、Vue Router 和 Element Plus 依赖，不增加身份类第三方依赖。
- 将认证表单状态与 API 调用封装在 auth store，视图只负责输入、展示与导航。
- 所有控件有明确 label、`autocomplete`、键盘可用焦点状态、禁用/提交中反馈和可读错误提示。
- 统一视觉变量、表单样式、按钮状态和移动端断点；登录与注册作为同一认证体验，避免页面风格分裂。
- 保留演示账号便捷入口，但将其标识为开发/演示辅助，不替代普通登录流程。

## 测试与验收

后端测试验证：静态前端资源与 history fallback 在构建产物存在时由 `main.py` 提供；API 路由不会被 SPA fallback 吞掉；注册始终产生客户身份，重复用户名和无效输入被拒绝。

前端测试验证：注册 payload、注册成功后的自动登录与跳转、注册失败展示、未登录重定向、各角色落地页、401 会话清除，以及同源 API 默认路径。最终执行 Python 相关测试、`npm test`、`npm run build`，并以 `python main.py` 启动后验证登录、注册和不同角色跳转。

## 非目标

- 不实现员工自助注册、邮箱/短信验证、密码找回、Refresh Token 轮换或 SSO。
- 不将 Vite 生产服务器作为运行依赖。
- 不重构既有各角色业务工作台，除非为认证接入、响应式一致性或构建托管所必需。
