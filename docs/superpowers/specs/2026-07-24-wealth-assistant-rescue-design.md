# 财富助手与画像同步修复设计

## 目标

修复财富助手的回复排版、同一用户跨重新进入的会话记忆、固定聊天工作区、理财经理快捷示例、风评到画像的同步，以及客户本人咨询时的画像和持仓上下文注入。

## 范围与原则

- 会话历史按已认证用户 ID 隔离；任何客户端传入的用户 ID 都不用于授权或历史归属。
- 只恢复当前用户自己的最近会话。理财经理的客户业务上下文不与其个人会话历史混用。
- 风评结果以 MySQL 画像主表为权威来源；Redis 只作可失效缓存。
- 不引入不受控的 HTML 渲染。助手回复的 Markdown 必须先转义不可信 HTML。
- 保留现有统一 `/api/chat`、JWT 认证和数据响应信封，避免创建平行对话入口。

## 设计

### 1. 持久化会话与页面恢复

服务端增加受 JWT 保护的历史查询接口，按认证用户 ID 查询最近一次会话的归档消息，按创建时间正序返回。消息响应包含会话 ID、角色、内容和时间；查询不得接受可覆盖身份的用户 ID。

前端会话 store 以认证用户 ID 为键保存会话状态。进入财富助手时：先读取内存状态；若为空则请求历史接口，恢复最近会话 ID 和消息。发送新消息时继续使用恢复的会话 ID；若服务端生成了会话 ID，则替换本地 ID。退出、刷新和重新进入后，同一用户能看到此前消息；切换用户不会读取到其他用户历史。

现有归档流程改为在当前轮回复前完成用户与助手消息的持久化，或使用等价的单次持久化方法，保证重新进入后立刻可读。Redis 短期记忆继续用于当前会话，MySQL 归档用于页面恢复和跨会话召回。

### 2. 客户上下文与助手记忆

统一路由分派投顾 Agent 时，若认证角色为客户、消息中未提取客户 ID，则把认证用户 ID 作为客户上下文传入。理财顾问、客户经理等员工角色不自动回退到自身 ID，仍须使用已选择或已解析的客户 ID。

投顾 Agent 用该客户 ID 读取画像摘要、持仓和历史偏好。因此“评估我的持仓”“我上一个问题是什么”等客户问题能基于真实画像、持仓和归档历史回答，而不是进入“缺少具体投资组合”的兜底回复。

### 3. 回复渲染

新增前端纯函数，将助手回复安全渲染为受限 Markdown：段落、空行、标题、无序/有序列表、加粗、行内代码、引用和分隔线。先转义 HTML，再生成受控标签；用户消息始终以纯文本展示。消息卡片对助手内容使用渲染结果，并提供适当的行高、段距、列表缩进和窄屏换行。

### 4. 财富助手与业务操作界面

财富助手页面保留单个页面标题，移除聊天卡片内重复的标题文案。聊天卡片采用视口高度布局：页头和输入区不滚动，消息区 `flex: 1` 且 `min-height: 0` 后独立滚动；页面本身不因对话增长出现滚轮。小屏保持自然纵向布局。

业务操作页的快捷示例改成与财富助手一致的两列提示卡片。点击卡片仅写入输入框，不直接执行操作；手机端降为单列。

### 5. 风评与画像一致性

风评提交完成后在同一事务中更新风险评估记录、画像风险等级/评分、画像快照中的风险字段、风险偏好标签和评分历史，随后提交事务并失效画像缓存。Neo4j 同步继续是可失败的后续同步，不能影响 MySQL 的权威更新。

前端风险问卷组件在提交成功后发布包含客户 ID 和新风险结果的更新事件。画像页、理财经理工作区和财富助手订阅该事件：当前展示的是同一客户时重新加载画像或更新上下文，确保无需手动刷新。

### 6. 测试与验收

- 前端：验证受限 Markdown 转义和结构化输出；验证会话 store 能按用户恢复、隔离并复用服务端会话 ID；验证快捷示例与固定聊天区的关键 DOM/CSS 约束。
- 后端：验证历史接口只能返回认证用户归档；验证客户身份进入投顾时传入认证用户 ID，员工身份不会错误回退；验证风评提交更新画像快照、标签、历史且清除缓存。
- 回归：修复并保留画像 API 对最小画像对象的序列化契约测试；运行前端单测、前端构建和目标 Python 单测。

## 错误处理

- 历史读取失败时显示空会话和可重试提示，不阻塞发送新消息。
- 历史中损坏的单条消息跳过，不影响其余消息恢复。
- Markdown 解析异常时回退为已转义的纯文本。
- 风评提交失败时不发布画像更新事件，也不显示成功状态。
- 客户没有画像或持仓时，助手明确说明缺少的数据类型；已有画像或持仓时不得使用该兜底文案。
